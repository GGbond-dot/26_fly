## 7. 异常与风险点

### 1. bag 文件不可用于复盘

这是本次数据分析最大限制。没有 `/height`、`/laser_array/ground_height`、`/fine_data`、`/circle_area_ratio` 的逐帧 rosbag 数据，就无法严谨复算测高公式、占比排序或视觉误差收敛。

### 2. 启动早期 TF 缺失

日志中有 11 条 `map -> laser_link` / `map <- laser_link` lookup 失败，集中在启动早期。cartographer/robot_state_publisher 启动后任务仍继续运行并完成。

结论：这是启动时序问题，不是本次任务失败原因；但可以考虑 mission 等待 TF 可用后再开始发 takeoff，减少无效等待和早期告警。

### 3. RViz 崩溃

`rviz2` 因 rockchip GL 驱动问题退出：

```text
rviz2 process has died, exit code -11
libGL error: failed to load driver: rockchip
```

这不影响 mission 主流程，但在飞行机上建议不要默认启动 RViz，避免占资源和引入噪声。

### 4. `pillar 3` 视觉采样不足

`pillar 3` 进入 align 后约 4 秒跳过，日志为：

```text
[pillar 3] 采样柱子帧不足 (6)，本柱跳过
```

这说明视觉有效采样不足，建议下次重点查：

- `/fine_data` 是否持续发布；
- `/circle_area_ratio` 是否有数据；
- 目标是否离画面中心太远或边框/铁片识别不稳定；
- align 超时和最少采样帧门槛是否太紧。

### 5. 点阵高度存在异常值

mission 完成后，uart 打印：

```text
Current height from flight controller: 32767
```

这像是无效值/溢出值。若 `/height` 直接参与测高，应在上位机侧过滤 `32767` 这类哨兵值，避免污染后续测高窗口。

## 8. 对下一次试飞的建议

1. 修正/确认运行模式：如果目标是“只测高不抓取”，应确保日志显示 `measure_only_mode=true`，且 `traverse_only_mode=false`。
2. 保留完整 rosbag 目录：至少包含 `metadata.yaml` 和所有 `.db3` 文件；飞行结束后等 recorder 正常退出再拷贝。
3. 下次重点录并复盘 `/height`、`/laser_array/ground_height`、`/fine_data`、`/circle_area_ratio`、`/mission_step`、`/target_position`。
4. 若柱高真实值远高于 27.7/51.0/77.6 cm，优先复查 `cam_offset_dx_cm=-5.1` 方向和 `LASER_MOUNT_DIFF_CM=9.0`。
5. 处理 `32767` 点阵高度：至少在测高采样逻辑中忽略明显无效值。
6. 飞行 launch 中可临时关闭 RViz，减少板端图形栈崩溃噪声。
